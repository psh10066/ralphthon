# 랄프톤 유저스토리

> Deep Interview 스펙 (16라운드, 모호도 10%) 기반 작성

---

## 1. 역할 그룹

| 그룹 | 역할 | 설명 | 주요 관심사 |
|------|------|------|-------------|
| **주 사용자** | 웹 첫 방문자 | 인증 없이 에센스 체험하는 익명 사용자 | 호기심, 즉시 결과, 저장 유도 |
| **주 사용자** | 가입 사용자 | 텔레그램 연동 완료한 사용자 | 축적, 패턴 발견, 연속 사용 |
| **주 사용자** | 텔레그램 사용자 | 텔레그램 봇에서 대화하는 사용자 | 가벼운 입력, 빠른 응답, 웹 연결 |
| **시스템** | AI 엔진 | Claude API (에센스/대화/축적) | 일관성, 품질, 톤 |

## 3. 핵심 유저스토리 선별

상태 변경 + 복잡성 + 영향도 기준:

| # | 스토리 | 선별 이유 |
|---|--------|----------|
| US2 | 사진 1~5장 유연 업로드 → 에센스 분석 | 상태 변경 + 사진 수 유연화 복잡성 |
| US6+US7 | Lazy Auth (저장 → 텔레그램 딥링크 → 마이그레이션) | 상태 변경 + 높은 복잡성 + 영향도 |

## 8. 검증 강제화 규칙

<HARD-GATE>
아래 규칙은 CLAUDE.md의 QA 절차와 동일한 강제력을 가진다.
어떤 실행 모드(autopilot, ralph, team, ultrawork)든 이 규칙을 따라야 한다.
</HARD-GATE>

### 8-1. 검증 순서 강제

```
Phase 1: 구현 완료
    ↓
Phase 2: 자동 Eval (에센스/대화/축적 골든 셋)
    ↓ PASS 필수
Phase 3: Playwright MCP 로컬 QA (7-1 ~ 7-7 전체, 52개 항목)
    ↓ PASS 필수
Phase 4: Vercel 배포
    ↓
Phase 5: Playwright MCP 배포 QA (배포 URL로 전체 재검증)
    ↓ PASS 필수
Phase 6: Playwright MCP 모바일 QA (390px 뷰포트)
    ↓ PASS 필수
Phase 7: 완료 선언
```

### 8-2. 검증 증거 요구

| 주장 | 필요한 증거 |
|------|------------|
| "US 검증 완료" | 해당 US의 모든 검증 항목에 대한 Playwright 스크린샷 + 콘솔 로그 |
| "갭 항목 통과" | G1~G7, S1~S4 각각의 Playwright 실행 로그 |
| "전체 QA 통과" | 52개 검증 항목 전체의 PASS/FAIL 리포트 |
| "배포 QA 통과" | 배포 URL에서 52개 항목 재실행 결과 |
| "모바일 QA 통과" | 390px 뷰포트에서 전체 페이지 스크린샷 |

### 8-3. 실패 시 처리

- 검증 항목 1개라도 FAIL → 수정 후 **해당 Phase를 처음부터 재실행**
- Phase 3 (로컬 QA) 실패 → Phase 4 (배포) 진행 금지
- Phase 5 (배포 QA) 실패 → 수정 → 재배포 → Phase 5 재실행
- "사소한 이슈라 넘어가자" → 금지. 전부 수정.

각 검증 항목은 TaskCreate로 태스크를 생성하고, Playwright 실행 증거와 함께 완료 처리해야 한다.
