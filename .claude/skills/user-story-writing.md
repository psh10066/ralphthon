# 유저스토리 작성 스킬 (User Story Writing Skill)

## 스킬 정의
독립적이고 완전한 유저스토리를 작성하고, 체계적인 변형 케이스 탐색을 통해 놓치기 쉬운 요구사항을 모두 발견하는 능력

## 입력 (Input)
- 기획 목표 및 범위
- 대상 사용자 역할 정보
- 현재 사용자 행동 패턴

## 출력 (Output)
- 역할별 구조화된 유저스토리
- 핵심 유저스토리의 변형 케이스 분석 결과
- 테스트 가능한 수준의 구체적 스토리

---

## 스킬 실행 단계

### 1단계: 역할 그룹 식별 및 분류
**목적**: 기능을 사용할 모든 사용자 역할을 체계적으로 파악

#### 역할 그룹 분류 체계
| 그룹 | 설명 | FinOps 예시 | 주요 관심사 |
|------|------|-------------|-------------|
| **주 사용자** | 기능의 직접적 수혜자 | 카드 소지자, 관리자 | 편의성, 효율성 |
| **관련 사용자** | 기능의 간접적 영향 받는 자 | 세무 담당자, 임원 | 정확성, 투명성 |
| **시스템 운영자** | 기능의 관리/운영 담당자 | GoWid 어드민, CS팀 | 관리 편의성, 오류 대응 |
| **외부 관련자** | 시스템 외부의 이해관계자 | 카드사, 회계사 | 연동성, 호환성 |

#### 역할 식별 체크리스트
- [ ] 기능을 직접 사용하는 모든 역할을 식별했는가?
- [ ] 기능으로 인해 영향받는 간접 사용자도 고려했는가?
- [ ] 시스템 운영 관점에서 필요한 역할이 있는가?
- [ ] 외부 시스템과의 연동이 있다면 관련 역할을 포함했는가?

### 2단계: 기본 유저스토리 작성
**목적**: 각 역할별로 핵심 니즈를 명확한 유저스토리로 표현

#### 유저스토리 작성 템플릿
```markdown
| # | As a | I want to | So that |
|---|------|-----------|---------|
| US1 | [구체적 역할] | [하고 싶은 것] | [이유/가치] |
```

#### 좋은 유저스토리 기준 (INVEST)
- **Independent (독립적)**: 다른 스토리와 독립적으로 구현 가능
- **Negotiable (협상가능)**: 구현 방법에 대해 논의 가능
- **Valuable (가치있음)**: 사용자에게 명확한 가치 제공  
- **Estimable (추정가능)**: 개발 공수 추정 가능
- **Small (작음)**: 한 스프린트에 완료 가능한 크기
- **Testable (테스트가능)**: 완료 여부를 명확히 판단 가능

#### 역할별 유저스토리 예시

##### 예시 1: 다중 용도 설정 기능
```markdown
### 관리자 (Admin)
| # | As a | I want to | So that |
|---|------|-----------|---------|
| US1 | 관리자 | 용도별로 필요한 세부 입력 항목을 설정할 수 있다 | 직원들이 용도에 맞는 구체적 정보를 입력하도록 할 수 있다 |
| US2 | 관리자 | 여러 용도에 공통 항목을 한 번에 설정할 수 있다 | 반복 작업 없이 효율적으로 관리할 수 있다 |

### 카드 소지자 (Card Holder)  
| # | As a | I want to | So that |
|---|------|-----------|---------|
| US3 | 카드 소지자 | 용도 선택 시 해당 용도의 필수 정보가 무엇인지 바로 알 수 있다 | 빠뜨리는 정보 없이 한 번에 제출을 완료할 수 있다 |
| US4 | 카드 소지자 | 이전에 입력한 유사한 내역의 정보를 재사용할 수 있다 | 반복되는 입력 작업을 줄일 수 있다 |

### 세무 담당자 (Accounting)
| # | As a | I want to | So that |
|---|------|-----------|---------|
| US5 | 세무 담당자 | 용도별로 수집된 구조화된 정보를 확인할 수 있다 | 회계 처리에 필요한 정보를 빠르게 파악할 수 있다 |
```

#### 기본 유저스토리 품질 체크리스트
- [ ] 각 스토리가 "As a - I want to - So that" 형식에 맞는가?
- [ ] "So that" 부분이 구체적인 가치나 이유를 제시하는가?
- [ ] 역할별로 해당 역할의 관점에서 작성되었는가?
- [ ] 모든 스토리가 INVEST 기준에 부합하는가?

### 3단계: 핵심 유저스토리 선별
**목적**: 변형 케이스 탐색이 필요한 핵심 스토리 식별

#### 핵심 스토리 선별 기준
- **상태 변경**: 설정, 생성, 수정, 삭제 등 시스템 상태를 변경하는 스토리
- **복잡성**: 여러 단계나 조건이 포함된 스토리  
- **영향도**: 다른 기능이나 사용자에게 영향을 미치는 스토리
- **위험도**: 실패 시 큰 문제가 될 수 있는 스토리

#### 핵심 스토리 예시 (위 예시에서)
- US1: "용도별로 필요한 세부 입력 항목을 설정할 수 있다" ← **상태 변경 + 복잡성**
- US2: "여러 용도에 공통 항목을 한 번에 설정할 수 있다" ← **복잡성 + 영향도**

### 4단계: 변형 케이스 탐색 (6축 분석)
**목적**: 핵심 유저스토리에 대해 놓치기 쉬운 변형 케이스를 체계적으로 발견

#### 6가지 변형 축
1. **수량 (Quantity)**: 하나가 아닌 여러 개, 최대값 제한
2. **방향 (Direction)**: 생성뿐만 아니라 수정, 삭제, 되돌리기
3. **결과 (Outcome)**: 성공뿐만 아니라 실패, 취소, 부분 성공
4. **권한 (Actor)**: 본인 것뿐만 아니라 타인 것, 권한 부족
5. **타이밍 (Timing)**: 즉시가 아닌 예약, 동시 접근, 임시 저장
6. **상태 (State)**: 깨끗한 상태가 아닌 기존 데이터 존재, 연결된 데이터

#### 변형 케이스 탐색 템플릿
```markdown
### US1 변형 케이스 분석: "용도별로 필요한 세부 입력 항목을 설정할 수 있다"

| 변형 축 | 질문 | 해당? | In/Out | 새로운 유저스토리 |
|--------|------|-------|--------|------------------|
| 수량 | 여러 용도에 한 번에 설정하려면? | Y | In | US1-1: 여러 용도 일괄 설정 |
| 수량 | 용도당 항목을 최대 몇 개까지? | Y | In | US1-2: 항목 개수 제한 설정 |
| 방향 | 설정을 수정하려면? | Y | In | US1-3: 기존 항목 수정 |
| 방향 | 설정을 삭제하려면? | Y | In | US1-4: 항목 삭제 |
| 결과 | 설정 저장이 실패하면? | Y | In | US1-5: 오류 상황 처리 |
| 권한 | 권한 없는 사람이 시도하면? | Y | In | US1-6: 권한 검증 및 차단 |
| 타이밍 | 임시 저장 후 나중에 완성하려면? | N | Out | v2.0에서 고려 |
| 상태 | 이미 설정된 용도에 추가하려면? | Y | In | US1-7: 기존 설정 위에 추가 |
```

#### 변형 케이스 도출 결과 예시
```markdown
### 추가된 유저스토리 (변형 케이스)
| # | As a | I want to | So that |
|---|------|-----------|---------|
| US1-1 | 관리자 | 여러 용도를 선택해서 동일한 항목을 한 번에 설정할 수 있다 | 반복 작업 없이 효율적으로 설정할 수 있다 |
| US1-2 | 관리자 | 용도당 설정 가능한 항목 개수 제한을 확인할 수 있다 | 너무 많은 항목으로 사용자 혼란을 방지할 수 있다 |
| US1-3 | 관리자 | 이미 설정된 항목의 내용을 수정할 수 있다 | 설정 실수를 처음부터 다시 하지 않고 바로잡을 수 있다 |
| US1-4 | 관리자 | 필요없어진 항목을 삭제할 수 있다 | 사용자에게 불필요한 입력 부담을 주지 않을 수 있다 |
| US1-5 | 관리자 | 설정 저장 실패 시 명확한 오류 메시지를 볼 수 있다 | 문제를 파악하고 해결할 수 있다 |
| US1-6 | 일반 사용자 | 권한이 없는 설정 화면 접근 시 적절한 안내를 받을 수 있다 | 혼란 없이 올바른 경로로 이동할 수 있다 |
| US1-7 | 관리자 | 기존 설정이 있는 용도에 새 항목을 추가할 수 있다 | 기존 설정을 유지하면서 필요한 항목만 보완할 수 있다 |
```

### 5단계: 유저스토리 우선순위 및 구현 순서 결정
**목적**: 개발 효율성과 사용자 가치를 고려한 구현 순서 결정

#### 우선순위 결정 기준
| 우선순위 | 기준 | 예시 |
|----------|------|------|
| **P0 (필수)** | 기본 기능 동작에 필수 | US1: 기본 항목 설정 |
| **P1 (중요)** | 사용성에 큰 영향 | US1-1: 여러 용도 일괄 설정 |
| **P2 (개선)** | 편의성 향상 | US1-3: 항목 수정 |
| **P3 (부가)** | 에지 케이스 또는 고급 기능 | US1-5: 오류 처리 |

#### 의존성 고려사항
- 기본 기능 → 확장 기능 순서
- 데이터 생성 → 데이터 수정/삭제 순서  
- 핵심 플로우 → 예외 상황 처리 순서

### 6단계: 최종 유저스토리 정리 및 검증
**목적**: 완성된 유저스토리 세트의 완결성과 일관성 확인

#### 최종 유저스토리 정리 템플릿
```markdown
## 6. 유저스토리

### 관리자 (Admin)
| # | As a | I want to | So that | 우선순위 |
|---|------|-----------|---------|----------|
| US1 | 관리자 | 용도별로 필요한 세부 입력 항목을 설정할 수 있다 | 직원들이 용도에 맞는 구체적 정보를 입력하도록 할 수 있다 | P0 |
| US1-1 | 관리자 | 여러 용도를 선택해서 동일한 항목을 한 번에 설정할 수 있다 | 반복 작업 없이 효율적으로 설정할 수 있다 | P1 |

### 변형 케이스 검토
- [x] 핵심 US에 대해 [6축 변형 케이스 탐색] 완료
- [x] 도출된 변형 케이스를 In Scope / Out of Scope 분류 완료
- [x] In Scope 변형 케이스를 추가 유저스토리로 작성 완료
```

#### 완결성 검증 체크리스트
- [ ] 모든 식별된 역할에 대한 유저스토리가 작성되었는가?
- [ ] 핵심 유저스토리에 대한 변형 케이스 탐색이 완료되었는가?
- [ ] CRUD (생성, 조회, 수정, 삭제) 관점에서 빠진 기능이 없는가?
- [ ] 에러 처리 및 예외 상황에 대한 스토리가 포함되었는가?

#### 일관성 검증 체크리스트  
- [ ] 모든 스토리가 동일한 형식과 톤앤매너로 작성되었는가?
- [ ] 스토리 간에 모순되는 요구사항은 없는가?
- [ ] 우선순위와 의존성이 논리적으로 설정되었는가?
- [ ] 각 스토리가 측정 가능한 완료 기준을 가지는가?

## 자주 발생하는 실수와 해결법

### 실수 1: 너무 큰 유저스토리 작성
**증상**: "관리자로서 지출 관리 시스템을 사용하고 싶다"처럼 모호하고 큰 스토리
**해결**: 구체적인 행동 단위로 분해
```
나쁜 예: "관리자로서 지출 관리 시스템을 사용하고 싶다"
좋은 예: "관리자로서 승인 대기 중인 이용내역 목록을 확인할 수 있다"
```

### 실수 2: 기술적 구현 방법을 스토리에 포함
**증상**: "REST API를 통해" 같은 구현 방법이 유저스토리에 포함됨
**해결**: 사용자 관점에서 '무엇을', '왜' 하고 싶은지만 기술
```
나쁜 예: "REST API를 통해 데이터를 조회하고 싶다"  
좋은 예: "이용내역 목록을 실시간으로 확인하고 싶다"
```

### 실수 3: 변형 케이스 탐색을 형식적으로만 수행
**증상**: 6축을 모두 "N" 처리하거나 실제 요구사항과 동떨어진 케이스만 도출
**해결**: 실제 사용자 시나리오를 기반으로 현실적인 변형 케이스 도출

### 실수 4: 우선순위 없이 나열만 함
**증상**: 모든 스토리가 동일한 중요도로 나열되어 개발 순서 결정 어려움
**해결**: P0~P3 우선순위와 의존성을 명확히 표시

## 품질 체크리스트

### 기본 유저스토리 품질 체크리스트
- [ ] 모든 스토리가 "As a - I want to - So that" 형식인가?
- [ ] 각 스토리가 독립적으로 구현 및 테스트 가능한가?
- [ ] "So that" 부분이 구체적인 가치를 제시하는가?
- [ ] 모든 역할 그룹에 대한 스토리가 포함되어 있는가?

### 변형 케이스 탐색 품질 체크리스트
- [ ] 핵심 유저스토리(상태 변경 스토리)를 모두 식별했는가?
- [ ] 6축 변형 분석을 체계적으로 수행했는가?
- [ ] 도출된 변형 케이스가 현실적이고 의미 있는가?
- [ ] In/Out Scope 판단이 명확한 근거를 가지는가?

### 전체 유저스토리 세트 품질 체크리스트  
- [ ] CRUD 관점에서 빠진 기능이 없는가?
- [ ] 에러 처리 및 예외 상황이 고려되었는가?
- [ ] 스토리 간 의존성과 우선순위가 명확한가?
- [ ] 개발팀이 구현 가능한 수준으로 구체적인가?

### 테스트 가능성 체크리스트
- [ ] 각 스토리의 완료 여부를 명확히 판단할 수 있는가?
- [ ] QA팀이 테스트 시나리오를 도출할 수 있는 수준인가?
- [ ] 사용자 수용 테스트 기준이 명확한가?
