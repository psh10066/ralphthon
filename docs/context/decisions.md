# 핵심 결정 로그

| # | 결정 | 대안 | 왜 이쪽을 골랐나 |
|---|------|------|-----------------|
| 1 | 입구 = 호기심(사진) | FOMO(글 저장) | FOMO와 사진 업로드는 다른 감정. 섞으면 퍼널이 꼬임 |
| 2 | wow = 에센스 언어 | UI 테마 동적 생성 | HTML 테스트 결과 시각적 차이 체감 약함. "손 닿는 곳의 아늑함" 언어가 사람을 잡음 |
| 3 | 에센스 6축 (양감/질감/투명도/촉각/무게/온도) | 컬러+무드만 | 배민 디자이너의 에센스 개념. 양감/촉각 같은 물리적 축이 더 깊은 읽기를 만듦 |
| 4 | 메타대화 = 모든 입력의 즉시 반응 | 에센스 후 별도 단계 | 풍경 사진 하나에도 "왜?" 가 바로 와야 서비스의 본질 |
| 5 | 텔레그램 = 시작점, 웹 = 축적 | 웹앱만 | 대화는 메신저가 자연스러움. 축적/지형도는 넓은 화면이 필요 |
| 6 | 진단하지 않고 질문한다 | 성격 유형 분류 | "MBTI는 라벨, 이건 거울." 진단은 닫고, 질문은 연다 |
| 7 | 축적 = 지형도 시각화 | 리스트/타임라인만 | 레이어링 느낌. 어떤 영역이 두꺼운지 한눈에 |
| 8 | 시간 축 = "그때의 나" | 현재 상태만 | 변화사가 가치. "3개월 전엔 이랬는데 지금은 다르다"가 보여야 |
| 9 | Supabase (Postgres) | localStorage | 텔레그램+웹 멀티채널이면 서버 사이드 저장 필수. 텔레그램에서 대화하고 웹에서 보려면 공유 DB가 있어야 함 |
| 10 | "취향도 자기 이해의 한 축" | 메타대화가 유일한 본질 | 세션 29에서 연애 매칭까지 갔다 돌아옴. 취향 탐색 자체가 가치 |
| 11 | 에센스 = 결론이 아니라 대화의 시작 | 에센스가 "당신은 이런 사람" | 메타대화 코어 검토에서 발견. 에센스가 진단이 되면 MBTI와 같은 구조. "이렇게 읽어봤어요, 더 알고 싶어요" 태도 |
| 12 | 첫 질문 = 열린 호기심, 가설은 2턴부터 | 첫 턴에서 가설 질문 | 감각→심리상태 점프는 진단. "뭐가 끌렸어요?"가 먼저, 가설은 사용자 응답 후 |
| 13 | 공유 = 에센스 직후가 아니라 대화 후 | 에센스 결과에서 바로 공유 | 바로 공유하면 에센스가 "결론"이 됨. 발견이 생긴 뒤 프로필에서 공유 |
